# AI Workflow

## Which AI tools and models did you use, and for what?

I used Claude Sonnet 4.6 throughout the assessment in two distinct ways.

### Development Assistance (Claude.ai)

Claude was used selectively as a pair-programming assistant for bounded implementation tasks, including:

* Scaffolding ASP.NET Core boilerplate (dependency injection, controller skeletons and service registration)
* Generating initial CsvHelper configuration
* Drafting the first version of the system prompt
* Creating React component shells and TypeScript interfaces
* Producing an initial implementation roadmap

These outputs were treated as starting points rather than final solutions and were reviewed, modified and integrated manually.

### Runtime Inference

The running application uses Claude Sonnet (API model string `claude-sonnet-4-5`) to answer investor questions.

Importantly, the runtime model does not perform financial calculations. It receives a structured context generated by the backend and is responsible only for producing natural language responses tailored to the selected investor.

No other AI tools or coding agents were used during development.

# The Central Engineering Decision

The most important design decision was that the language model should never perform financial calculations.

Instead, every financial figure is calculated deterministically within the .NET backend before any request reaches the model.

This separation reflects two fundamentally different types of work:

Deterministic computation

* Portfolio value
* MOIC
* FX conversion
* Outstanding commitments
* Fees
* Distributions

These calculations have one correct answer and must remain reproducible.

Natural language generation

* Explaining portfolio performance
* Answering investor questions
* Personalising responses
* Simplifying financial terminology

These are exactly the tasks language models perform well.

The backend therefore computes every financial value first.

The model receives only verified portfolio context and converts those facts into natural language.

It never performs arithmetic over raw CSV data.

---

# Why This Separation Matters

Separating deterministic business logic from AI provides several important benefits.

## Reliability

Financial calculations are performed entirely within C# and were manually verified against ground-truth figures derived directly from the supplied CSV files.

The same investor inputs always produce the same financial outputs, because the model never touches the arithmetic.

## Verification

Ground-truth figures were derived by hand and used to validate the running application.

A regression test suite is the highest priority next step — without it, regressions would only surface at runtime rather than in an automated check.

## Security and Data Isolation

Investor filtering occurs before prompt construction.

Only the authenticated investor's data is included in the generated context, making it structurally impossible for the model to access another investor's portfolio.

## Auditability

The backend generates deterministic source mappings from the CSV repository.

The application includes file-level source citations beneath each response so users can see which datasets contributed to the generated answer.

Row-level citations were intentionally deferred as a future enhancement.

## Replaceable AI Layer

Because the model consumes structured context rather than raw business data, the AI provider is largely interchangeable.

Replacing Claude with another LLM would require minimal changes to the surrounding application.

---

# Engineering Decisions and Trade-offs

Several implementation approaches were intentionally rejected or modified during development.

## Rejected: Tool Calling

One option was exposing backend services as callable AI tools.

I rejected this approach because the entire investor context comfortably fits within the model's context window.

Generating the context deterministically before invoking the model produces a simpler, more predictable and more auditable architecture.

## Rejected: Streaming Responses

Streaming improves perceived responsiveness but provides little value if the underlying financial calculations are incorrect.

Within the assessment timebox I prioritised correctness over user experience enhancements.

Streaming remains the first production enhancement I would implement.

# Examples of Manual Engineering

Several AI-generated implementations required significant manual changes before they were suitable.

Examples include:

* Correcting the CSV delimiter from tab-separated to comma-separated after validating against the supplied dataset.
* Reworking the CsvHelper configuration to avoid shared static mapping conflicts.
* Correcting MOIC calculations to use contributed capital rather than total commitment.
* Grouping investments by company rather than deal, allowing multiple funding rounds to appear as a single investor position.
* Extending the `InvestorContext` model to include statement lines and live FX rates.
* Resolving runtime issues including CsvHelper mapping conflicts, prompt size limitations, missing context fields and API integration issues.

These changes required understanding both the business domain and the supplied dataset rather than simply accepting generated code.

# If I Had Another Eight Hours

With additional development time, I would prioritise the following work:

1. Automated regression test suite encoding the ground-truth figures as assertions, so regressions surface immediately rather than at runtime.
2. Full investor verification by executing portfolio generation across all 112 investors and validating against the supplied dataset.
3. End-to-end AI evaluation, automatically checking that generated responses remain consistent with deterministic portfolio calculations.
4. Streaming responses using Server-Sent Events.
5. Row-level source citations, allowing responses to reference the exact records used during calculation.
6. Portfolio visualisations, including valuation history charts for each investment.

# Reflection

AI significantly accelerated implementation by reducing the amount of boilerplate code that needed to be written manually.

However, the success of the solution depended far more on engineering judgement than on code generation.

The architectural boundary between deterministic financial calculations and AI-generated language shaped every major design decision.

Throughout the assessment, I treated AI as an implementation assistant rather than the system's decision-maker.

Business logic, financial correctness, testing, debugging and verification remained deterministic responsibilities owned by the application itself.

That separation is what makes the solution reliable, explainable and suitable for handling investor-facing financial information.
